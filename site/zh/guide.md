# 使用指南

Lore Forge 是一个 AI 辅助的叙事设计框架。它通过结构化访谈、知识管理、角色设计、叙事规划、对话创作和一致性审查，帮助你构建并维护一个活生生的**故事圣经**。

你扮演**创意总监**的角色。AI 读取框架指令，选择合适的智能体，并遵循相应的工作流。你回答问题并做最终决定。

## 本指南适合谁

- **故事创作者** — 想用 Lore Forge 构建故事世界的写作者、游戏设计师、跑团主持人
- **读者** — 想浏览和阅读已完成故事项目的任何人

## 环境准备

### 检查运行需求

Lore Forge 需要你的 AI 平台提供 **文件读写**、**跨会话记忆**、**内容搜索** 三项能力。

让 Agent 一键安装：

```
读取 REQUIREMENT.zh-CN.md。检测当前平台并安装最低需要的 MCP 服务器。
验证每个工具是否可用，报告当前已具备哪些能力。
```

然后重启平台。详见[运行需求](/zh/requirements)页面的手动安装和各平台详细说明。

### 打开仓库

克隆仓库并在 AI 助手中打开：

```bash
git clone https://github.com/immane/lore-forge.git
cd lore-forge
```

Lore Forge 支持 **Codex、Claude Code、OpenCode、Cursor、GitHub Copilot agents、Gemini CLI** 等 AI 编码和文档智能体。打开仓库时大多数工具会自动读取 `AGENTS.md`。

如果你的助手没有自动加载仓库指令，请从以下开始：

```
读取 AGENTS.md 并确认 Lore Forge 已就绪可用。
```

## 快速开始

### 创建新项目

```
创建一个新的 Lore Forge 项目。

项目名称：月光
类型：叙事型 RPG
灵感来源：To The Moon、Finding Paradise
主题：遗憾、原谅、记忆
目标时长：6-8 小时
目标平台：PC
```

AI 会在 `projects/active/moonlight/` 下搭建项目结构，然后开始概念发现访谈。

### 继续已有项目

```
继续「月光」项目。
找出影响最大的缺失 Story Bible 信息，然后向我提出下一个问题。
```

### 创建角色

```
为「月光」创建主角 Luna。
使用 Character Therapist 工作流，开始角色发现访谈。
```

### 规划章节或路线

```
为「月光」根据当前 Story Bible 规划章节结构。
定义幕、章节分配、依赖关系图以及逐场景大纲。
```

### 编写场景

```
为「月光」生成第一章的场景定义。
使用 Narrative Designer 工作流，并保持与 Story Bible 正典一致。
```

### 编写对话

```
为「月光」编写第一章开场场景的对话。
使用已建立的角色档案、声音签名、潜台词和知识边界。
```

### 运行审计

```
对 projects/active/moonlight/ 运行完整的 Lore Forge 一致性审计。
报告关键矛盾、警告和建议修复方案，并附上文件引用。
```

### 导出

```
projects/active/moonlight/ 下的 Story Bible 已完成并通过审计。
导出 JSON Event Graph、Godot Dialogic 时间线、小说手稿 Markdown 和剧本参考文件。
```

如果你使用 OpenCode，`opencode.json` 还定义了项目创建、继续和审计的快捷命令。

## 完整工作流

以下是从空白文件夹到游戏或手稿就绪导出的完整流程。每一步都可以用自然语言调用。

### 阶段 1 — 概念发现

**Lore Architect** 智能体通过访谈提取世界观的基石。

```
创建一个新的 Lore Forge 项目并开始概念发现访谈。

项目名称：月光
类型：叙事型 RPG
灵感来源：To The Moon、Finding Paradise
主题：遗憾、原谅、记忆
目标时长：6-8 小时
目标平台：PC
```

Lore Architect 会问以下结构化问题：

- **情感核心** — "玩家在故事结束后应该带着怎样的感受？"
- **类型与基调** — "它在乐观—悲观谱系上处于什么位置？"
- **主题锚点** — "这个故事探索的核心问题是什么？"
- **世界规则** — "是否存在魔法体系、先进科技或超自然元素？"
- **结局设计** — "故事应该有哪些结局？每个结局如何让人觉得值得？"
- **内容边界** — "需要避免哪些主题、基调或内容类型？"

你的答案会写入：

- `story/vision.md` — 情感核心、承诺、基调、核心问题
- `story/themes.md` — 主要主题、子主题、母题地图
- `story/emotional_core.md` — 情感强度曲线、关键节拍
- `story/ending_design.md` — 结局清单、条件、回报
- `knowledge/canon.md` — 世界事实、唯一事实来源
- `knowledge/rules.md` — 魔法、科技、社会、经济规则
- `knowledge/glossary.md` — 世界内术语词典

预计会有 20–50 个问题，分多个会话进行。你可以随时暂停和恢复。

### 阶段 2 — 角色发现

**Character Therapist** 智能体以心理学深度发现你的角色阵容。

```
为「月光」创建主角 Luna。
使用 Character Therapist 工作流，开始角色发现访谈。
```

每个角色会定义：

- **核心创伤** — 是什么伤害了他们？何时？如何应对？
- **欲望金字塔** — 表面欲望 → 深层欲望 → 真正需求
- **恐惧阶梯** — 轻微焦虑 → 主要恐惧 → 存在性恐惧
- **防御机制** — 如何保护自己？
- **自我欺骗** — 对自己撒了什么谎？
- **声音签名** — 说话模式、词汇、口头禅、沉默模式
- **关系地图** — 与其他角色的双向关系
- **成长弧** — 故事开始 vs 结束时是谁？

每个角色输出到 `characters/{name}.md`。

```
现在创建反派，名字叫 Sol。
同时创建 Luna 的导师角色和她的童年好友。
```

### 阶段 3 — 时间线构建

**Narrative Designer** 构建世界层面的编年史。

```
为「月光」构建世界时间线和角色时间线。
从已确认的故事文件和角色文件中填充 Story Bible 时间线。
```

这会创建：
- **世界纪元** — 标志性事件、社会状态、关键人物
- **故事时间线** — 章节级编年史，含世界日期
- **角色时间线** — 出生、成长事件、故事入场/退场
- **交汇点** — 角色线索交汇之处

输出：`story/timeline.md`

### 阶段 4 — 记忆图谱

Narrative Designer 绘制**主观记忆**——角色记住了什么、何时记起、以及记忆是否可靠。

```
为「月光」构建记忆图谱。
追溯每个主要角色的记忆链。
```

记忆图谱不同于时间线——时间线是客观事实，记忆是主观的。角色对事件的记忆可能和实际发生的不同，这种差距创造了戏剧张力。

输出：`memories/_index.md`

### 阶段 5 — 叙事设计

Narrative Designer 将原材料组织成章节和场景。

```
为「月光」根据当前 Story Bible 规划章节结构。
定义幕、章节分配、依赖关系图以及逐场景大纲。
```

产出：
- **幕结构** — 三幕或五幕，含幕间事件
- **张力曲线** — 跨所有章节的戏剧张力映射
- **依赖关系图** — 章节前置要求和并行轨道
- **章节详情** — 梗概、视角、情感弧、所需场景

输出：`story/chapters/_outline.md`（视觉小说为 `story/routes/_outline.md`）

然后生成场景：

```
为「月光」第一章生成所有场景定义。
```

每个场景包含：戏剧功能、张力位置、情感转折、视角、信息流和伏笔整合。

### 阶段 6 — 对话创作

**Dialogue Writer** 赋予每个角色独特的声音。

```
为「月光」编写第一章开场场景的对话。
为每一行注释情感状态、潜台词和关系张力。
```

输出是带注释的对话：

```markdown
**Luna** [状态：焦虑，带着希望]
{潜台词：在试探 Sol 是否还记得——她不想直接问}
"你还会想起那个夜晚吗？"

**Sol** [状态：防备，不以为意]
{潜台词：记得清清楚楚，但不想让她得逞}
"哪个夜晚？夜晚有很多。"
```

输出：`dialogue/01-opening.md`

### 阶段 7 — 一致性审查

**Lore Auditor** 在确定为正典之前交叉引用所有模块。

```
对 projects/active/moonlight/ 运行完整的一致性审计。
报告所有违规、警告和建议。
```

检查项包括：

| 检查 | 验证内容 |
|-------|---------|
| 时间线 | 事件 B 在事件 A 引用它之前不会发生 |
| 角色知识 | 没有角色知道未被告诉的信息 |
| 声音一致性 | 每个角色的对话符合其声音签名 |
| 规则合规 | 没有内容违反 `knowledge/rules.md` |
| 伏笔 | 每个伏笔有回报；每个回报有伏笔 |
| 结局可达 | 所有预设结局仍然可达 |

严重级别：**严重**（必须修复）、**高**（阻碍正典化）、**中**（建议性）、**低**（风格建议）。

可以随时运行针对性审计：

```
为「月光」对角色 Luna 运行针对性审计。
```

### 阶段 8 — 导出

Story Bible 通过审计后，导出为生产就绪格式。

```
projects/active/moonlight/ 下的 Story Bible 已完成并通过审计。
导出为：
1. JSON Event Graph
2. Godot Dialogic 时间线
3. 小说手稿 Markdown
4. 剧本格式
```

输出：
- `exports/json/` — 供游戏引擎使用的结构化 JSON
- `exports/godot/` — 兼容 Dialogic 的时间线文件
- `exports/novel/` — 大纲导出、散文草稿、手稿 Markdown
- `exports/screenplay/` — Fountain 格式剧本

## 迭代

Lore Forge 是为持续迭代设计的，而非一次性生成：

```
1. 访谈 → Story Bible 增长
2. 编写新内容（角色、场景、对话）
3. Lore Auditor 检查一致性
4. 修复问题，如有空白则重新访谈
5. 重复直至 Story Bible 完成
6. 导出
```

## 在本站阅读项目

一旦项目构建并导出完成，你就可以在本站浏览它们：

- 前往**[项目列表](/zh/projects)**查看所有可用的故事世界
- 点击任意项目探索其完整的故事圣经——角色、地点、知识和导出内容
- 在每个项目的**小说**部分阅读小说章节
